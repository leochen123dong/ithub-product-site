## 二、平台架构与安全（platform.html）

### 2.1 开场（建议 1 分钟）

> 在讲具体功能之前，我们先看底座。一个平台好不好，底座决定了上层的一切。
>
> ITHub 基于 **.NET 10.0 企业级架构**构建。这是微软最新的长期支持版本（LTS），经过了大规模并发数据负载验证，达到电信级可靠性。ITHub 的架构设计围绕四大基础能力展开，确保平台在任何企业环境下都能可靠、安全、弹性地运行。

### 2.2 四大基础能力

#### 01 — 全球分布式弹性拓扑

> 这是 ITHub 能管大规模的关键。

- **Master Poller 架构**：区域采集枢纽，中心统一调度。从微型分支（几台设备）到跨国云环境（上万台设备）无缝扩展
- **VRF 网络隔离**：每个 Poller 支持专用 VRF 网络，不同租户的数据在网络层面绝对隔离。对 MSP 场景尤其重要——你管着 20 个客户的网络，客户 A 绝对看不到客户 B 的数据
- **弹性扩展**：单虚拟 Poller 支持多客户，全球分布式部署

#### 02 — 全栈异构集成

> 企业的 IT 环境往往是"万国牌"——Cisco 的交换机、华为的路由、Palo Alto 的防火墙、VMware 的虚拟化、AWS 的云。传统方案需要给每个厂商搞一套监控，数据格式不统一。

- ITHub 作为企业数据总线，**原生管理 100+ 厂商设备**，**10+ 采集协议**
- 把碎片化指标标准化为"单一事实来源"
- 客户不需要关心底层是 SNMP 还是 REST API 还是 WMI，ITHub 统一采集、统一呈现

#### 03 — 零信任电信级核心

> 对金融行业、政府客户来说，这一点尤其重要。

- 基于 .NET 10.0 构建，经过大规模并发数据负载验证
- **原生 TACACS+ 审计**：记录每一条网络设备命令——谁在什么时间对哪台设备执行了什么命令，全部有迹可循
- **加密密码保险箱**：运维人员通过 SSO 临时获取设备权限，全程不知道实际密码，杜绝密码泄露风险
- 满足最严格的金融合规标准

#### 04 — 企业服务管理 (ESM)

> ITHub 不只是 IT 部门的工具。

- 从传统 ITSM 演进为**统一企业服务枢纽**
- 可编程 AI 原生工作流可以扩展到 **HR**（员工入职/离职流程）、**设施**（会议室预订/门禁管理）、**财务**（费用审批/采购流程）、**安全**（安全事件响应）等部门
- 打破部门间的数据孤岛，一个平台服务全公司

### 2.3 安全保障体系

> 安全不是附加功能，而是 ITHub 的核心基因。从认证、授权、审计到数据保护，安全贯穿平台的每一个环节。

逐个讲解八个安全能力卡片：

**SSO 单点登录**
> 支持 OAuth 2.0、OIDC、SAML 三大协议，与企业现有的 AD/LDAP/Azure AD 无缝集成。员工用公司账号一键登录，不需要额外记忆密码。

**RBAC 角色控制**
> 细粒度的角色权限管理。可以按模块（谁能看监控、谁能看工单）、数据范围（只能看自己区域的设备）、用户组（网络组、安全组、管理组）三个维度定义访问权限。支持 40+ 种受保护对象类型，层级化权限体系支持继承。

**多租户架构**
> 不同客户或部门的数据、配置和访问权限完全独立。如果你是 MSP 管多个客户，或者大企业的不同事业部需要独立管理，多租户架构都能满足。

**密码保险箱**
> 这是很多客户特别感兴趣的功能。运维人员需要登录设备做操作时，通过 SSO 临时获取权限，全程不知道实际密码。操作结束后权限自动回收。从根本上解决了"运维人员掌握大量设备密码"的安全隐患。

**3 次失败锁定**
> 连续 3 次登录失败自动锁定账户，防暴力破解。简单但有效。

**会话审计与录屏**
> 运维操作全程审计，支持操作录屏回放。万一出了安全事故，可以完整回放操作过程，追溯责任。对等保合规、ISO 27001 审计都是刚需。

**TACACS+ 命令审计**
> 原生 TACACS+ 协议支持，记录每一条网络设备命令。不只是"谁登录了"，而是"谁在什么时间执行了 show running-config"还是"修改了 ACL 规则"，颗粒度到命令级别。

**SSL/TLS 全链路**
> Poller 与中心平台之间、浏览器与服务端之间，所有通信全部 SSL/TLS 加密，防止数据在传输过程中被截获。

**双因子认证 (2FA)**
> 支持一次性密码（OTP）双因子认证，用于 Web 登录和敏感资源访问的二次验证。

## 三、GEMMB 全局监控（gemmb.html）

### 3.1 开场（建议 1 分钟）

> GEMMB 全称是 Global Enterprise Monitoring & Management Board——全局企业管理监控看板。
>
> 它提供的是**全域可观测性**，覆盖企业 IT 基础架构的每一个层面。从网络设备到公有云，从安全防火墙到数据库，所有遥测数据汇聚在统一看板中。
>
> 几个关键数字：**5 大监控领域、100+ 技术厂商集成、10+ 监控采集协议、100K+ 设备纳管能力**。

### 3.2 五大监控领域

#### 01 — 网络与 SD-WAN

> 网络是 IT 基础设施的命脉，GEMMB 提供深度的遥测采集，覆盖企业路由、交换、无线控制器和 SD-WAN 编排器。

**SNMP 主动轮询**
> 支持 SNMP v2c/v3 协议，状态/统计/阈值三维监控，厂商无关设计。不管是 Cisco、Juniper、华为还是 H3C，统一采集、统一呈现。

**ICMP 可用性检测**
> 实时连通性检测与延迟监控，支持 Trace Route 路径追踪。不只是告诉你"通了没通"，还能看到"路径上哪一跳出了问题"。

**NetFlow / sFlow 流量分析**
> 多维度流量分解：Top Talkers（谁在用带宽）、Top Applications（什么应用在消耗带宽）、Top Protocols（什么协议占比最大）。这对排查"网络慢"的问题特别有用——到底是视频会议占了带宽，还是有人在下载大文件？

**SD-WAN API 集成**
> 这是 ITHub 的差异化亮点。原生 API 对接 **Viptela vManage、Cisco Meraki、VeloCloud、Aruba** 控制器，不只是监控 SD-WAN 隧道状态，还能拿到应用级别的体验数据。

覆盖厂商：Cisco、Cisco Meraki、Juniper、Huawei、H3C、Aruba、Viptela 等。

#### 02 — 安全与防火墙

> 安全设备的监控和安全日志的管理。

**Syslog 事件采集**
> 安全日志加密 Relay 转发，支持合规冷备存储 180 天。满足等保 2.0 对日志留存的要求。

**告警关联分析**
> 跨安全设备关联告警，自动去噪和分级，精准推送高危事件。比如防火墙报了一个"疑似入侵"，IDS 也报了一个"异常流量"，ITHub 能自动关联判断这是同一个安全事件。

**Splunk 集成**
> 安全告警实时集成到 GEMMB 看板，统一告警视图。如果客户已经在用 Splunk 做安全分析，不需要切到另一套系统，GEMMB 直接把 Splunk 的告警拉过来。

覆盖厂商：Palo Alto、FortiGate、Cisco IDS/IPS、Sangfor（深信服）、Hillstone（山石）等。

#### 03 — 计算与存储

**虚拟化深度监控**
> VMware vSphere API 深度集成，自动发现并监控虚拟机、主机、集群。不需要手动添加——只要 vCenter 里有的 VM，ITHub 自动发现、自动纳管。

**硬件带外管理**
> Dell iDRAC、HPE iLO、Cisco UCS 带外管理接口直接采集硬件状态。服务器的 CPU 温度、硬盘 SMART 状态、电源冗余状态、风扇转速，都能通过带外接口直接拿到，不依赖操作系统。

覆盖：VMware vSphere、Cisco UCS、Dell iDRAC、HPE iLO、NetApp、PureStorage 等。

#### 04 — 公有云与数字体验 (DEM)

**公有云 API 监控**
> 原生 REST API 对接 **AWS、Azure、阿里云、华为云、腾讯云**，自动发现云资源。不需要在云上装 Agent，直接通过 API 拉取 EC2 实例状态、RDS 数据库性能、S3 存储用量等指标。

**数字体验监控 (DEM)**

> 这对排查"灰色故障"特别有用——用户说"系统很慢"，但你从内部监控看一切正常。这种问题往往是 ISP 链路或 CDN 节点的问题，ThousandEyes 可以从路径级别定位到具体是哪一段出了问题。

#### 05 — 数据库与 OS Agent

### 3.3 深度监控引擎详解

> 页面上列出了五种核心监控引擎。这里要强调的是，ITHub 的监控不是简单的"Ping 一下看看通不通"，每种引擎都经过大规模生产环境验证：

**SNMP 监控引擎**
- 高度可配置的轮询引擎，厂商无关设计
- 状态、统计、阈值三维监控——不只看"UP/DOWN"，还看性能趋势
- 灵活实体模型，自动发现设备层级与依赖关系
- 内置远程排障工具：SNMP 实时查询、Ping、Trace Route
- 详细数据分析导航树，支持"忙时分析"（定位高峰时段的性能瓶颈）
- 支持 SNMP v1/v2c/v3，Trap 告警接收

**HTTP 监控引擎**
- 网站 / Web 服务 / Web 应用的健康检查与性能监控
- 可定制脚本引擎，轮询多种内容类型（不只检查 HTTP 状态码，还能检查页面内容是否符合预期）
- HTTP 响应状态与性能统计（总响应时间、内容类型分解）
- **SSL 证书监控**：过期提醒、名称不匹配告警、过时协议检测（比如还在用 TLS 1.0 的站点会被标记）

**DNS 监控引擎**
- 支持 A、MX、NS、URI、CNAME 等所有主流记录类型
- 查询结果与预配置值自动比对，异常即触发告警
- DNS 组管理，支持动态 IP 地址设备

**NetFlow / sFlow 流量分析**
- 多维度网络流量可视化分析
- Top Talkers、Top Listeners、Top Conversations
- Top Applications、Top Protocols、Top TOS
- NetFlow Relay 中继功能，数据可转发至第三方系统（比如 PRTG、SolarWinds）

**Syslog 日志管理**
- 分布式 Poller 高密采集 Syslog，支持全文检索与定制化过滤
- 基于 Syslog 自动生成告警（比如收到 "BGP neighbor down" 自动创建告警）
- 日志采集存储与全文检索
- C# 脚本定制化过滤规则
- **Syslog Relay 中继**：支持按客户/设备精细化转发控制，加密转发至合规冷备存储区，满足等保 2.0 规定的 180 天原始日志存储要求

### 3.4 Smart Monitoring 智能监控

> 这是 ITHub 减少无效工单的关键机制，也是跟开源监控方案最大的区别之一。

传统监控的问题：一条链路抖动 3 秒，瞬间产生 50 条告警、创建 50 张工单。运维人员被淹没在告警海洋中，真正的问题反而被忽略。

ITHub 的 Smart Monitoring 三步走：
1. **异常进入"观察"状态**：非正常告警不立即创建工单，而是先进入观察期
2. **观察期内自动判断**：如果恢复正常 → 自动关闭，不产生工单；如果超时未恢复 → 按状态设定严重等级，正式创建工单
3. **有效过滤噪声**：偶发性抖动告警被过滤，只有持续性故障才创建工单

> 效果：告警风暴抑制 90%，MTTR 降至分钟级，告警准确率 90%+。

### 3.5 Configure Harvest（配置自动收割）

> 配置管理是安全合规的基础。很多安全事件的根因不是"被攻击了"，而是"配置被改了自己不知道"。

- **灵活脚本引擎**：支持通过 Telnet、SSH 或 SNMP+TFTP 采集设备配置
- **历史版本比对**：配置文档自动保存并与历史版本逐行比对，清晰展示变更差异——哪一行加了、哪一行删了、哪一行改了，一目了然
- **定时调度执行**：可按日定时调度执行，防止配置漂移，确保合规基线
- **未授权变更报警**：发现不在变更管理流程中的配置变更时立即报警

### 3.6 网络拓扑发现与根因分析

> 这个功能在实际运维中非常实用。

- 自动发现被管设备的轮询拓扑与依赖关系，动态可视化展示
- 集成服务台进行根因分析：**多台设备不可达时，自动关联为同一事件生成一张工单**
- 比如核心交换机宕机，下挂的 20 台接入交换机全部不可达——传统监控会创建 21 张工单，ITHub 通过拓扑分析只会创建 1 张，指向根因（核心交换机）
- **告警风暴抑制 90%**

### 3.7 分布式 Poller 架构

> ITHub 的监控不是"从一个中心点去轮询所有设备"，而是分布式架构。

- **多租户 VRF 隔离**：每个 Poller 支持多租户部署，不同客户/部门之间的监控数据在网络层面绝对隔离
- **SSL 加密通信**：Poller 与中心平台之间的所有通信均通过 SSL/TLS 加密
- **弹性拓扑扩展**：从微型分支到跨国云环境无缝扩展，Master Poller 架构支持全球分布式部署
- **远程集中管理**：无需在每个节点安排运维人员，内置看门狗自动恢复软件故障

**Poller 部署能力**：
- 支持 Windows、Linux 和 Docker 部署
- 按客户、区域、位置或设备级别灵活分配
- 基于角色的 Poller 管理，支持委托部署
- 可配置 Ping 静默时间、SNMP 轮询间隔、HTTP 轮询间隔

### 3.8 服务器 Agent 深度管理

> 通过轻量级 Windows / Linux Agent 实现服务器环境的深度监控与远程管理。

**Windows / Linux Agent**
- 轻量级部署（约 100MB 内存，极低 CPU 占用）
- 通过 Web 门户集中配置，远程下发
- 灵活脚本化监控引擎：状态/统计/阈值监控与告警生成
- 全局默认配置 + 单项覆盖，自定义时间段阈值

**Windows 远程桌面**
- 即时（Just-in-Time）远程桌面访问——不需要提前开 RDP 端口
- 连接前自动加入远程桌面用户组，连接结束后自动移除权限
- 可选加入本地管理员组
- **与密码保险箱集成**——运维人员全程不知道服务器密码

**Windows 桌面共享**
- 通过 Server Agent 作为主机实现安全桌面共享
- 仅查看或交互控制模式，自定义访问权限级别
- 支持安全桌面（UAC）查看与交互

**Windows 事件日志监控**
- 灵活选择监控的事件日志和事件级别
- 事件 ID 过滤（包含/排除/范围）
- 事件汇总：同窗口内相同事件合并计数
- 事件过滤脚本，自定义告警文本

**Shell 脚本与系统更新**
- 直接在 Web 端执行 Shell / PowerShell 脚本，无需 SSH 登录
- Windows Update 过期更新告警，自动安装更新 / 手动检查下载

## 四、CMDB 配置管理（cmdb.html）

### 4.1 开场（建议 1 分钟）

> CMDB——Configuration Management Database，配置管理数据库。
>
> ITHub 的 CMDB 不是简单的资产台账。很多企业的 CMDB 就是一份 Excel 表，记录着"这台服务器叫什么、IP 是多少、谁负责"。这种 CMDB 的问题是：数据很快过时、无法跟监控系统联动、无法支撑变更影响分析。
>
> ITHub 的 CMDB 是一个**从数据采集到业务消费的完整价值链**。四个关键数字：**11 种数据类型、16+ 关系类型、6 维数据质量、20+ 消费场景**。

### 4.2 四层架构

> 从上到下四层，每层解决一个问题：

1. **数据采集层**：数据从哪来？——Excel 批量导入、API 同步、外部数据库、GEMMB Poller 自动采集、自动发现
2. **CI 建模层**：数据怎么组织？——CI Templates 定义类型、Property Definitions 定义属性、11 种数据类型
3. **关系引擎**：数据之间什么关系？——Relationship Definitions 定义关系、CI Extension 跨平台拓扑
4. **消费层**：数据怎么用？——ITSM 集成、AI 自然语言查询、REST API 报表导出

### 4.3 CI 建模引擎

**11 种数据类型**——覆盖了 IT 资产管理中几乎所有的字段需求：

| 数据类型 | 说明 | 典型用途 |
|---------|------|---------|
| String | 字符数据 | 设备名称、序列号 |
| Multi Line String | 多行字符串 | 描述、备注 |

### 4.4 关系模型

> CMDB 的价值不只是"记录资产"，更重要的是"记录资产之间的关系"。有了关系，才能做影响分析、根因定位。

四种关系类型：
- **Parent-Child 父子级**：品牌 → 服务器（Dell 品牌下有哪些服务器型号）
- **Depends On 依赖关系**：应用 → 数据库（这个应用依赖哪个数据库实例）
- **Contains 包含关系**：机柜 → 服务器（这个机柜里装了哪些设备）
- **Connected To 连接关系**：交换机端口 → 服务器（这台服务器接在交换机的哪个口上）

> 举个例子：核心交换机 A 故障了。如果有连接关系和依赖关系的记录，ITHub 可以秒级算出"交换机 A 下面连了哪些服务器，这些服务器上跑了哪些应用，这些应用影响了哪些业务"。这就是关系拓扑的价值。

### 4.5 CI Extension（监控集成）

> 这是 ITHub CMDB 区别于其他 CMDB 产品的核心差异——CI 不只是静态记录，还跟 GEMMB 监控系统深度关联。

三种扩展类型：
- **Device**：CI 关联到 GEMMB 监控设备（在 CMDB 里看到这台设备的实时监控状态）
- **Device Group**：CI 关联到设备组（比如"北京数据中心所有核心交换机"）
- **Contact**：CI 关联到联系人（设备出问题时一键找到负责人）

> 效果：在监控设备页面可以直接查看关联的 CI 属性（厂商、序列号、保修期、负责人），在 CMDB 页面也可以看到设备的实时运行状态。两个系统不是割裂的。

### 4.6 六维度数据质量治理

> 基于 DAMA 国际数据质量管理框架。CMDB 最大的挑战不是"建起来"，而是"数据准确"。一个数据不准确的 CMDB 比没有 CMDB 更危险——因为它会误导决策。

| 维度 | 治理能力 | 具体手段 |
|------|---------|---------|
| **准确性** | 确保数据正确 | Excel 导入数据校验、IP 地址冲突自动检测、API 数据源验证 |
| **完整性** | 确保数据齐全 | 必填字段强制校验、CI 属性填充率统计、缺失数据自动发现 |
| **一致性** | 确保多处数据一致 | CMDB ↔ ITSM 双向同步、CI Extension 监控关联、跨系统数据一致性保障 |
| **时效性** | 确保数据新鲜 | 到期监控（60/30/1 天）、自动分级告警 + 创建工单、合同/证书/许可证到期追踪 |

| **合规性** | 确保符合规范 | 变更审计追踪、多租户数据隔离、RBAC 访问控制 |

## 五、ITSM 服务管理（itsm.html）

### 5.1 开场（建议 1 分钟）

> ITSM 模块遵循 ITIL 最佳实践设计，覆盖企业 IT 服务管理的核心流程需求。
>
> 市面上 ITSM 系统很多——ServiceNow、BMC Remedy、Jira Service Management。ITHub 的 ITSM 有什么不同？最大的不同是：**它不是孤立系统**，而是跟 GEMMB 监控、CMDB 配置管理深度联动的。
>
> 监控发现告警 → 自动创建工单 → 工单关联 CI → 拓扑分析影响范围 → 智能分配给最低负载的工程师 → SLA 计时器精确管控 → 处理完成自动沉淀知识库。**这才是真正的端到端闭环**。
>
> 关键数字：**4 大 ITIL 流程、100K+ 工单容量验证、<200ms P95 查询响应、P1-P4 SLA 等级管控**。

### 5.2 四大 ITIL 核心流程

**事件管理 (Incident)**
> 目标：快速恢复服务，最小化业务中断影响。

- 自动/手动创建工单，告警直接关联
- P1-P4 优先级分级，SLA 计时器自动启动
- 拓扑根因分析：多台设备故障合并为一张工单（避免告警风暴导致的工单洪水）

**问题管理 (Problem)**
> 目标：深度分析反复出现的事件，找到并消除根本原因。

- 事件关联分析，识别反复出现的故障模式（比如"每个月第一个周一这台存储都会出问题"）
- 根因分析与解决方案追踪
- 知识库沉淀，防止同类问题再次发生

**变更管理 (Change)**
> 目标：标准化变更流程，降低变更风险，确保所有变更有迹可循。

- 变更请求审批流程，CI 变更自动触发 CR

| 模式 | 行为 | 适用场景 |

### 6.3 三大 AI 应用场景

#### 01 — Smart RMA 闭环

> 硬件故障的闭环处理，从发现到换件全自动。

- **故障自动识别**：AI 引擎分析遥测数据，自动判定硬盘、内存、电源、风扇等故障类型——不需要人去查看 SNMP Trap 里到底报了什么
- **SLA 合同匹配**：自动查询设备保修/SLA 合同信息，确认 RMA 资格——不需要人去翻合同
- **物流自动触发**：生成 RMA 工单并触发备件物流流程，同步通知各方——不需要人打电话联系供应商
- **效果**：精确故障预测与自动化 RMA，**减少备件库存 15%**。硬件故障秒级闭环。

#### 02 — 威胁语义分析

> 安全告警的最大问题是"误报太多"和"关联不起来"。

- **攻击链重建**：跨源关联安全日志，自动还原攻击路径和时间线，一图看清攻击全貌
- **告警去噪**：语义分析自动过滤误报和重复告警，只推送真正需要关注的高危事件
- **效果**：MTTR 降至分钟级，告警准确率 90%+

#### 03 — 多场景智能解析

**Syslog 秒级根因定位**
- 对 Syslog 进行秒级语义分析，自动从海量日志中定位故障根因
- 生成诊断快照并秒级派发工单
- 传统方式需要运维工程师逐条看日志，AI 直接告诉你"第 3847 行日志是关键线索"

**邮件智能开单 (Event Log AI Agent)**
- 理解自然语言邮件内容，自动提取故障描述、影响范围、紧急程度
- 自动填充工单属性
- 完全消除人工分拣环节——以前需要一个 L1 工程师每天花 2 小时分拣邮件，现在 AI 自动完成
